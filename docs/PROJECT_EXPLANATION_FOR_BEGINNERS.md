# 1. 项目整体介绍

## 1.1 这个项目是什么？

想象你是一个城市共享单车的运营经理。每天早上你到公司，最想知道的事情可能是：

- 现在全市有多少辆单车可用？
- 哪个站点没车了（空车风险）？
- 哪个站点桩位全满还不了车（满桩风险）？
- 过去 24 小时用车量有什么规律？
- 系统有没有异常告警？
- 能不能直接问 AI "哪个区域用车压力最大？"

**本项目就是一个帮你回答以上所有问题的 Web 系统。**

它不是一个简单的"数据展示页面"，而是一个**多功能的共享单车运营分析平台**。你可以把它理解为：

> 一个给共享单车运营团队用的"智能指挥大屏"——既能看实时数据，又能查历史趋势，还能让 AI 帮你分析运营问题，甚至能自动从真实城市数据源抓取最新数据。

## 1.2 系统有哪些页面？分别做什么？

| 页面 | 路径 | 作用 |
|---|---|---|
| **首页大屏** | `/` | 一屏总览全市运营状态，像作战室的大屏幕 |
| **区域分析** | `/regions` | 按地理区域查看站点分布和占用率排行 |
| **站点管理** | `/stations` | 查看所有站点的列表、搜索、筛选 |
| **单站详情** | `/stations/SZ-SO-S001` | 查看某个具体站点的实时状态和历史曲线 |
| **告警中心** | `/alerts` | 集中查看所有异常告警（空车、满桩、离线等） |
| **趋势分析** | `/trends` | 查看全市过去 24 小时的用车趋势图表 |
| **ETL 管理** | `/etl` | 管理真实数据源的抓取、查看抓取日志和调度状态 |
| **AI 助手** | `/ai` 或右侧抽屉 | 用自然语言向 AI 提问，获取数据分析和运营建议 |
| **系统设置** | `/settings` | 切换主题、调整刷新频率、查看 AI 服务状态 |

> **路径**就是你浏览器地址栏里看到的那串地址，比如 `http://localhost:3000/regions`

## 1.3 系统最终解决了什么问题？

用一句话概括：

> **让运营人员用最低的学习成本，最快地掌握整座城市的共享单车运行态势，并在问题发生前就收到预警。**

具体解决以下 7 个场景：

1. **查看城市运行状态**：打开首页，一眼看到总车辆数、总桩位数、平均占用率、风险站点数。
2. **查看站点分布**：地图上用不同颜色标记所有站点，绿色正常、黄色高占用、红色空车/满桩、灰色离线。
3. **查看车辆供需**：哪些站点车太少（空车风险）、哪些站点车太多还不了（满桩风险）。
4. **查看实时告警**：系统自动检测异常，生成告警卡片，点击可跳转到地图定位。
5. **查看真实数据抓取状态**：系统每 5 分钟自动从纽约/旧金山等城市抓取真实 GBFS 数据，ETL 页面显示抓取是否成功。
6. **使用 AI 询问运营情况**：不用写代码、不用查数据库，直接打字问 AI"哪个站点容量最大？""当前有多少个满桩风险？"
7. **数据持久化与历史回溯**：所有数据存入数据库，可以回看过去任意时刻的站点状态。

## 3.1 前端技术栈

前端就是你在浏览器里看到的界面——按钮、图表、地图、文字颜色、布局等等。

| 技术 | 作用（大白话） |

| **localStorage** | 浏览器本地存储。关掉网页再打开，数据还在。本项目用来保存用户的主题偏好、设置选项、AI 聊天记录。 |

## 3.2 后端技术栈

后端就是运行在服务器上的程序，负责：接收前端的请求、查数据库、处理数据、调用 AI、返回结果给前端。

| 技术 | 作用（大白话） |
|---|---|
| **FastAPI** | Python Web 框架。负责提供 REST API（就是一组 URL，前端可以通过 HTTP 请求来获取数据）。特点是速度快、自动生成交互式 API 文档。 |
| **Python** | 编程语言。后端全部用 Python 写，因为它语法简洁，数据处理能力强。 |
| **SQLModel** | 数据库 ORM 工具。让你不用手写 SQL 语句，直接用 Python 对象操作数据库。比如 `session.exec(select(Station))` 就等同于 `SELECT * FROM station`。 |
| **Pydantic** | 数据校验库。定义数据应该长什么样（比如"接口返回的站点数据必须有 name 和 capacity"），自动校验前端传来的数据是否合法。 |
| **APScheduler** | 定时任务调度器。每 5 分钟自动执行一次"从真实数据源抓取数据"的任务，不需要人工干预。 |
| **httpx** | HTTP 客户端。用来向外部的 GBFS 数据源发请求，拉取真实城市的共享单车数据。 |
| **OpenAI SDK** | Kimi / Moonshot 提供的兼容 SDK。虽然名字叫 OpenAI SDK，但它可以配置不同的 base_url，用来调用国产 Kimi API。 |

## 3.3 数据库

数据库就是存放所有数据的地方。本项目目前使用 **SQLite**，但设计上可以平滑迁移到 **PostgreSQL + PostGIS**。

### 为什么是 SQLite？

SQLite 是一个"文件型数据库"——整个数据库就是一个 `.db` 文件，不需要安装额外的数据库服务器。适合：
- 开发阶段快速验证
- 单机部署演示
- 数据量不大时性能足够

### 为什么是 PostgreSQL + PostGIS（后续扩展）？

PostgreSQL 是一个企业级关系型数据库，PostGIS 是它的空间数据扩展。共享单车涉及：
- 大量站点（数万个）
- 经纬度坐标（空间查询："查找我附近 500 米内的站点"）
- 高并发读写（实时状态更新）

这些场景下 PostgreSQL + PostGIS 比 SQLite 更合适。

### 数据库有哪些表？每张表存什么？

| 表名 | 存储内容 | 关键字段 |
|---|---|---|
| `station` | 站点静态信息（名称、经纬度、容量、所属区域） | code, name, lat, lng, capacity, region_id, source_system |
| `cur_station_status` | 站点实时状态（当前可用车数、可用桩数、占用率） | station_id, num_bikes, num_docks, occupancy_rate, status |
| `fact_station_snapshot` | 历史快照（每隔一段时间记录一次全量站点状态，用于趋势分析） | station_id, ts, num_bikes, num_docks, occupancy_rate |
| `region` | 地理区域信息（城市分区） | code, name, city |
| `alert` | 系统告警记录（空车、满桩、离线等） | id, station_id, level, type, message, created_at |
| `data_source` | 数据源配置（Mock / Citi Bike / Bay Wheels 等） | code, name, provider, gbfs_url, enabled, city |
| `gbfs_feed` | GBFS 数据源的子 feed 列表（station_information, station_status 等） | source_id, feed_name, url |
| `fetch_log` | 数据抓取日志（每次抓取的时间、状态、耗时、记录数、错误信息） | id, feed_id, started_at, finished_at, status, records, duration_ms |

        │                    │
        │                    ▼
        │            ┌─────────────────────────┐
        │            │ fact_station_snapshot   │
        │            │    (历史快照)            │
        │            └─────────────────────────┘
        │
        ▼
┌─────────────┐     ┌─────────────────────┐
│data_source  │────▶│    gbfs_feed        │
│ (数据源配置) │1:N  │  (GBFS feed 列表)   │
└─────────────┘     └─────────────────────┘
        │
        │

- `region` 和 `station`：**一对多**（一个区域有多个站点）
- `station` 和 `cur_station_status`：**一对一**（一个站点只有一条实时状态）
- `station` 和 `fact_station_snapshot`：**一对多**（一个站点有很多条历史记录）
- `data_source` 和 `gbfs_feed`：**一对多**（一个数据源有多个 feed，比如 station_info 和 station_status）
- `fetch_log`：记录每一次抓取操作的结果

## 3.4 AI 部分

AI 不是前端直接调用的，而是走后端代理。为什么？

### 为什么不能前端直接调用 Kimi？

因为 API Key 是敏感信息。如果前端直接调用，Key 就会暴露在浏览器里，任何人"查看网页源代码"就能偷走你的 Key，拿去乱用。

### AI 的 Tool-Calling 是什么？

当用户问"查一下市民中心站点的状态"，AI 不能瞎编，必须查询真实数据。Tool-Calling 就是：

1. AI 发现需要数据，决定调用工具 `get_station_detail`
2. 后端执行这个工具函数，返回真实数据
3. AI 拿到数据后，生成最终回答

本项目 AI 可用的工具包括：

| 工具名 | 作用 |
|---|---|
| `get_station_detail` | 查询单个站点的实时状态 |
| `get_station_capacity_ranking` | 查询站点容量排行 |
| `get_region_summary` | 查询区域汇总数据 |

## 3.5 数据采集（ETL）

ETL 是 Extract（抽取）、Transform（转换）、Load（加载）的缩写。就是"把外部数据抓过来，整理一下，存进数据库"。

### 数据从哪里来？

本项目支持两种数据来源：

1. **Mock 数据**：开发阶段模拟的 150 个站点，3 个城市，用于快速验证功能。
2. **真实 GBFS 数据**：从公开数据源抓取。
   - Citi Bike（纽约）：`https://gbfs.citibikenyc.com/gbfs/gbfs.json`
   - Bay Wheels（旧金山）：`https://gbfs.baywheels.com/gbfs/gbfs.json`

### GBFS 是什么？

GBFS（General Bikeshare Feed Specification）是共享单车行业的通用数据标准。它规定了一个标准的 API 格式，让任何城市都可以按统一格式公开自己的单车数据。

GBFS 的核心文件包括：
- `gbfs.json` — 自动发现文件，列出了所有子 feed 的 URL
- `station_information.json` — 站点静态信息（名称、位置、容量）
- `station_status.json` — 站点实时状态（可用车数、可用桩数）

### ETL 流程

```
┌─────────────────┐
│ APScheduler     │  ← 每 5 分钟触发一次
│ 定时调度器       │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ etl_service     │
│ run_etl_for_    │  ← 查 data_source 表，获取数据源配置
│ source()        │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ gbfs_client     │  ← 用 httpx 请求 GBFS 接口
│ fetch_gbfs_     │
│ discovery()     │
└────────┬────────┘
         │
         ├──▶ 请求 station_information.json
         │
         ├──▶ 请求 station_status.json
         │
         ▼
┌─────────────────┐
│ 数据合并        │  ← 按 station_id 把 info + status 合并
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Upsert station  │  ← 更新/插入站点表
│ Upsert status   │  ← 更新实时状态表
│ Insert snapshot │  ← 插入历史快照表
│ Write fetch_log │  ← 记录抓取日志
└─────────────────┘
```

### Fallback 机制

如果真实数据抓取失败，系统不会崩溃：

```
真实数据最近 5 分钟有成功记录？
    ├── 是 → 优先用真实数据（只查 source_system = 'US-CITIBIKE' 等）
    └── 否 → 回退到 Mock 数据（查全部数据，含 source_system = 'mock'）
```

这样即使网络中断，首页大屏仍然能显示数据（Mock 兜底）。

# 5. 前后端如何协作：一条数据请求的完整旅程

理解前后端怎么配合，是理解整个系统的关键。下面用"打开首页"这个场景，一步步讲解数据是怎么从前端流到后端，再流回前端的。

## 5.1 场景：用户打开首页

### 第 1 步：浏览器请求页面

```
浏览器输入 http://localhost:3000/
         │
         ▼
    ┌─────────────┐
    │  Next.js    │  ← 前端框架渲染页面
    │  返回 HTML  │
    └─────────────┘
```

### 第 2 步：页面加载后，前端请求数据

页面加载完成后，前端需要获取：
- KPI 数据（总站点数、可用车辆等）
- 地图数据（所有站点的位置和状态）
- 趋势数据（过去 24 小时曲线）
- 风险站点列表
- 最近告警

这些请求是并发的（同时发出）：

```
前端浏览器
    ├── GET /api/dashboard/kpi          ──▶
    ├── GET /api/dashboard/stations     ──▶
    ├── GET /api/dashboard/trends       ──▶
    ├── GET /api/dashboard/risks        ──▶
    └── GET /api/dashboard/alerts       ──▶
                                          │
                                          ▼
                                    ┌─────────────┐
                                    │  FastAPI    │
                                    │  后端路由   │
                                    └──────┬──────┘
                                           │
                              ┌────────────┼────────────┐
                              ▼            ▼            ▼
                         ┌────────┐  ┌──────────┐  ┌──────────┐
                         │路由函数│  │  service │  │  数据库   │
                         │        │──│  业务逻辑 │──│  SQLite   │
                         └────────┘  └──────────┘  └──────────┘
                              │
                              ▼
                         返回 JSON
                              │
                              ▼
前端浏览器 ◀───────────────────┘
    │
    ├── 解析 KPI JSON，更新数字显示
    ├── 解析站点 JSON，渲染地图标记
    ├── 解析趋势 JSON，渲染 ECharts 折线图
    ├── 解析风险 JSON，渲染风险卡片列表
    └── 解析告警 JSON，渲染告警滚动条
```

### 第 3 步：自动刷新

首页不是"一次性加载"，而是每隔一段时间自动刷新数据：

```
TanStack Query 配置：
  - KPI：每 30 秒刷新
  - 地图：每 30 秒刷新
  - 告警：每 10 秒刷新

这样运营人员不需要手动刷新，数据自动保持最新。
```

## 5.2 场景：用户问 AI "哪个站点容量最大？"

这是一个更复杂的场景，涉及 tool-calling。

```
用户在前端输入："哪个站点容量最大？"
         │
         ▼
前端构建消息历史（最近 10 条 + 当前问题）
         │
         ▼
POST /api/ai/chat/stream
Body: { messages: [...] }
         │
         ▼
后端 routes_ai.py 收到请求
    ├── 注入系统提示词（告诉 AI 你是谁、能做什么）
    ├── 注入数据上下文（当前系统状态摘要）
    ├── 将完整消息发给 Kimi API
    │
    │   Kimi 思考后决定：需要调用 get_station_capacity_ranking 工具
    │
    ├── 收到 tool_calls 响应
    ├── 执行 get_station_capacity_ranking 函数（查数据库）
    ├── 将工具结果追加到对话历史
    ├── 再次发给 Kimi
    │
    │   Kimi 拿到数据后生成最终回答
    │
    └── 通过 SSE（Server-Sent Events）流式返回给前端
         │
         ▼
前端逐字显示 AI 的回答（打字机效果）
同时显示工具调用状态卡片（"正在查询站点容量排行..." → "完成"）
```

### SSE 是什么？为什么要流式返回？

SSE = Server-Sent Events，服务器向客户端推送事件。

普通请求：后端等 AI 全部想完了，一次性把整段话返回给前端。用户要等好几秒，屏幕空白，体验很差。

SSE 流式：AI 每生成一个字，就立即推送给前端。前端可以立即显示，用户看到"打字机效果"，感觉 AI 在实时思考。

```
普通模式：              流式 SSE 模式：
[等待 5 秒]            [AI 正在输入...]
[一次性显示整段话]      H
                       He
                       Hel
                       Hell
                       Hello
```

# 6. 每个功能页面的详细实现说明

## 6.1 首页大屏（`/`）

### 页面布局

```
┌─────────────────────────────────────────────────────┐
│  KPI 指标栏（6 个卡片）                              │
│  [总站点] [可用车辆] [可用桩位] [占用率] [空车] [满桩] │
├─────────────────────────────────────────────────────┤
│                                                     │
│          地图（占主要区域）                           │
│          [站点标记点：绿色/黄色/红色/灰色]            │
│                                                     │
├──────────────────┬──────────────────────────────────┤
│  24h 趋势图       │  区域排行                         │
│  [ECharts 折线图] │  [各区域平均占用率排序]            │
├──────────────────┴──────────────────────────────────┤
│  风险站点列表（空车/满桩）  │  最近告警                  │
│  [可点击跳转到地图定位]     │  [可点击定位]              │
└─────────────────────────────────────────────────────┘
```

### 数据流

1. **页面加载** → 同时发起 5 个 API 请求（KPI、站点、趋势、风险、告警）
2. **TanStack Query 管理** → 自动缓存、自动刷新、处理加载态
3. **KPI 数据** → `dashboard_service.get_kpi()` 聚合查询 station + cur_status 表
4. **地图数据** → `dashboard_service.get_stations_geojson()` 返回 GeoJSON 格式（地图库直接可用）
5. **趋势数据** → `dashboard_service.get_trends()` 查询 fact_station_snapshot 表按小时聚合
6. **风险数据** → `dashboard_service.get_risks()` 查询 cur_status 中 status 为 'empty_risk' 或 'full_risk' 的站点
7. **告警数据** → `dashboard_service.get_recent_alerts()` 查询 alert 表最近 N 条

### 为什么用 GeoJSON？

地图库（MapLibre GL）需要特定格式的数据才能显示标记点。GeoJSON 是一种标准的地理数据格式：

```json
{
  "type": "FeatureCollection",
  "features": [
    {
      "type": "Feature",
      "geometry": {
        "type": "Point",
        "coordinates": [114.0579, 22.5431]
      },
      "properties": {
        "name": "市民中心",
        "status": "normal",
        "occupancy_rate": 0.65
      }
    }
  ]
}
```

后端直接返回这种格式，前端地图库可以直接渲染。

## 6.2 区域分析（`/regions`）

### 功能

- 左侧：区域列表，显示每个区域的站点数量、平均占用率
- 点击区域：展开显示该区域下的所有站点
- 站点按占用率排序

### 实现

1. 后端 `dashboard_service.get_region_ranking()` 按 region_id 分组聚合
2. 计算每个区域的：站点数量、平均可用车数、平均可用桩数、平均占用率
3. 前端用折叠面板展示（点击展开/收起）

## 6.3 站点管理（`/stations`）

### 功能

- 网格卡片展示所有站点
- 每个卡片显示：名称、容量、可用车/桩、占用率、状态标签
- 搜索框：按名称模糊搜索
- 状态筛选标签：全部/正常/空车/满桩/离线

### 实现

1. 后端 `/api/stations` 返回分页列表（或全部列表）
2. 前端用 `useState` 管理筛选条件
3. 搜索和筛选在前端本地完成（数据量不大时更流畅）

## 6.4 单站详情（`/stations/[code]`）

### URL 中的 `[code]` 是什么？

这是 Next.js 的**动态路由**。`[code]` 是一个占位符，可以匹配任何站点编码：

- `/stations/SZ-SO-S001` → 显示"市民中心"站点
- `/stations/SZ-NS-S015` → 显示"科技园"站点

### 页面内容

1. **站点基本信息**：名称、编码、容量、经纬度、状态标签
2. **实时状态大数字**：可用车数、可用桩数、占用率（大字体显示）
3. **24 小时历史曲线**：用 ECharts 折线图展示过去 24 小时的可用车数变化
4. **历史告警**：该站点产生的所有告警记录

### 历史曲线数据来源

`fact_station_snapshot` 表每隔一段时间记录一次所有站点的状态。查单站历史就是：

```sql
SELECT ts, num_bikes, num_docks, occupancy_rate
FROM fact_station_snapshot
WHERE station_id = ?
  AND ts > datetime('now', '-24 hours')
ORDER BY ts
```

## 6.5 告警中心（`/alerts`）

### 告警是怎么产生的？

不是人工录入的，而是**系统自动检测**的。

后台有一个定时任务（或通过 ETL 流程），每次更新站点状态时检查：

| 条件 | 告警级别 | 告警类型 |
|---|---|---|
| 可用车数 = 0 | critical | 空车风险 |
| 可用桩数 = 0 | critical | 满桩风险 |
| 占用率 > 90% | warning | 高占用 |
| 占用率 < 10% | warning | 低占用 |
| 站点未上报状态（离线） | critical | 离线 |

### 页面功能

- 告警表格：时间、站点、级别、类型、消息
- 级别筛选：critical / warning / info
- 点击告警行 → 跳转到地图定位

## 6.6 趋势分析（`/trends`）

### 功能

用折线图展示全市过去 24 小时的：
- 可用车辆总数变化
- 可用桩位总数变化
- 平均占用率变化

### 实现

1. 后端按小时聚合 `fact_station_snapshot`：
   ```sql
   SELECT strftime('%Y-%m-%d %H:00:00', ts) as hour,
          SUM(num_bikes) as total_bikes,
          SUM(num_docks) as total_docks,
          AVG(occupancy_rate) as avg_occupancy
   FROM fact_station_snapshot
   WHERE ts > datetime('now', '-24 hours')
   GROUP BY hour
   ORDER BY hour
   ```
2. 前端用 ECharts 渲染三条折线

```

    │
    ▼

    │
    ▼

    ▼

    │

    │
    ▼

```

### 上下文策略

为了不让 AI "忘记"之前的对话，又不让请求太长，采用策略：

> 每次发送请求时，带上最近 10 条 user/assistant 消息 + 当前问题。

为什么是 10 条？
- 太少（比如 2 条）→ AI 容易忘记之前的上下文
- 太多（比如 50 条）→ 请求太长，耗 token，响应慢
- 10 条是平衡点，覆盖大约 5 轮对话（每轮 user + assistant = 2 条）

# 7. 数据库设计详解

## 7.1 为什么这样设计表？

共享单车的核心业务围绕"站点"展开。一个站点有：
- **静态属性**：名称、位置、容量（不常变）
- **动态属性**：当前可用车数、可用桩数（经常变）
- **历史记录**：过去每一时刻的状态（用于趋势分析）

所以拆成三张表：

| 表 | 存储什么 | 更新频率 |
|---|---|---|
| `station` | 静态信息 | 很少（新站点/站点改造时） |
| `cur_station_status` | 当前状态 | 每次 ETL 抓取时（5 分钟一次） |
| `fact_station_snapshot` | 历史快照 | 每次 ETL 抓取时追加 |

### 为什么静态和动态分开？

假设把静态信息和实时状态放在同一张表里：

```sql
-- 不好的设计
station (id, name, lat, lng, capacity, num_bikes, num_docks, ...)
```

每次抓取只更新了 `num_bikes` 和 `num_docks`，但 name/lat/lng/capacity 完全没变，却要更新整行。而且静态信息（name, lat, lng）查询频率和动态信息不同。

分开后：
- 查"全市站点列表" → 主要查 `station` 表
- 查"实时占用率" → JOIN `station` + `cur_station_status`
- 查"历史趋势" → 查 `fact_station_snapshot`

## 7.2 `fact_station_snapshot` 是怎么生成的？

每次 ETL 抓取完成后，系统会做一个"快照"：

```
ETL 完成时间：2025-01-15 14:05:00
对齐到整点：  2025-01-15 14:00:00

对每个站点，插入一条记录：
  station_id = 站点ID
  ts = '2025-01-15 14:00:00'
  num_bikes = 当前可用车数
  num_docks = 当前可用桩数
  occupancy_rate = 占用率
```

这样每小时一个"快照点"，连成线就是趋势图。

> 对齐到整点的好处：不同数据源的快照时间统一，方便对比。

## 7.3 `source_system` 字段的作用

`station` 表和 `cur_station_status` 表都有一个 `source_system` 字段，取值可能是：
- `mock` — Mock 数据
- `US-CITIBIKE` — Citi Bike 真实数据
- `US-BAYWHEELS` — Bay Wheels 真实数据

**为什么需要这个字段？**

系统支持同时存在多数据源：
- Mock 数据一直存在（开发/演示用）
- 真实数据抓取后也会入库

`source_system` 让系统能区分"这批站点来自哪个数据源"。

Dashboard 查询时：
- 如果最近 5 分钟有真实数据抓取成功 → 只查真实数据（`WHERE source_system != 'mock'`）
- 如果真实数据不可用 → 回退到 Mock 数据（查全部）

## 9.2 可能的答辩问题与回答

### Q1：为什么选择 FastAPI 而不是其他框架？

> "FastAPI 有三个优势：第一，自动生成 API 文档，开发调试方便；第二，原生支持异步，处理流式 AI 响应性能好；第三，Python 生态丰富，数据处理库多。"

### Q2：AI 是怎么知道当前系统数据的？

> "每次 AI 对话时，后端会把当前系统的实时数据（站点数、风险数、占用率等）打包成 JSON，作为上下文注入到系统提示词里。所以 AI 的回答是基于真实数据的，不会瞎编。"

### Q3：真实数据和 Mock 数据怎么切换？

> "Dashboard 查询时，后端会检查最近 5 分钟内是否有真实数据的抓取成功记录。如果有，就只查真实数据；如果没有，就回退到 Mock 数据。这个过程对用户是透明的。"

### Q4：如果我想加一个新的城市数据源，怎么做？

> "只需要三步：第一，在 ETL 页面添加新的数据源配置（填 GBFS URL）；第二，点击'立即抓取'测试；第三，如果抓取成功，APScheduler 会自动按 5 分钟间隔调度。不需要改代码。"

### Q5：这个系统能支撑多大的数据量？

> "目前用 SQLite，适合演示和中小规模数据。设计上所有 SQL 都是标准 SQL，可以平滑迁移到 PostgreSQL + PostGIS。PostGIS 支持空间索引和地理查询，适合数万站点级别的城市。"

# 10. 附录：常见术语解释

| 术语 | 解释 |
|---|---|
| **API** | 应用程序接口。前端通过 API 向后端请求数据，就像餐厅的服务员（API）把客人（前端）的点单传给厨房（后端），再把菜端回来。 |
| **REST** | 一种 API 设计风格。用 URL 表示资源，用 HTTP 方法（GET/POST/PUT/DELETE）表示操作。 |
| **JSON** | 一种数据格式。人类可读，机器易解析，是前后端交换数据的标准格式。 |
| **ORM** | 对象关系映射。让你用操作对象的方式操作数据库，不用手写 SQL。 |
| **SQL** | 结构化查询语言。用来和数据库对话的语言，比如 `SELECT * FROM station`。 |
| **SSE** | Server-Sent Events。服务器向浏览器推送实时数据，适合"打字机效果"的流式输出。 |
| **GBFS** | 共享单车通用数据标准。规定了共享单车数据应该怎么公开，全世界很多城市都用这个标准。 |
| **ETL** | 抽取-转换-加载。把外部数据抓过来、整理、存入数据库的过程。 |
| **KPI** | 关键绩效指标。比如"总站点数""平均占用率"这些核心数字。 |
| **GeoJSON** | 地理数据格式。用来在地图上标记点的标准格式，包含经纬度和属性信息。 |
| **Hooks** | React 的函数式编程概念。让函数组件也能拥有状态（比如 `useState`）和副作用（比如 `useEffect`）。 |
| **localStorage** | 浏览器内置的本地存储。关掉网页数据还在，但只能存字符串，容量约 5MB。 |
| **Pydantic** | Python 的数据校验库。定义数据"应该长什么样"，自动检查数据是否合法。 |
| **ORM** | Object-Relational Mapping。用 Python 类代表数据库表，用对象操作代替手写 SQL。 |
| **APScheduler** | Python 的定时任务库。像手机上的闹钟，到点就执行指定的函数。 |
| **Fallback** | 降级策略。主方案失败时，自动切换到备用方案。 |

3. **精装修**（M4-M7）：AI 助手、流式响应、真实数据、自动调度

如果你是零基础，建议按这个顺序去理解：
1. 先理解"这个项目是做什么的"（第 1、2 章）
2. 再理解"用了什么技术"（第 3 章）
3. 然后理解"代码怎么组织的"（第 4 章）
4. 接着理解"数据怎么流动的"（第 5 章）
5. 最后理解"每个页面怎么实现的"（第 6 章）

技术只是工具，理解**业务逻辑**（共享单车运营要解决什么问题）和**数据流**（数据从哪来到哪去）才是核心。
